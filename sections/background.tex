\section{Anycast}\label{sec:background-anycast}
The first RFC describing anycast (RFC~1546~\cite{rfc1546}) was published in 1993 and explains how anycast simplifies finding an appropriate server.
As service examples, they list DNS, FTP, and ``Archie'' (one of the first Internet search engines).
Nowadays, anycast is widely deployed for critical Internet infrastructure, such as the DNS~\cite{dns-anycast}, or by \glspl{cdn} to provide web services~\cite{CicaleseEtAl_CharacterizingIPv4Anycast_2015, GiordanoEtAl_FirstCharacterizationAnycast_}.

Anycast is the practice of making a service IP address (either IPv4 or IPv6) available in multiple physical locations~\cite{rfc4786}.
Such locations are referred to catchments or simply as \glspl{pop}.
% G: \glspl{pop} are also catchments. Not needed to introduce here.
For instance, when a user is located in Enschede trying to access a website that is anycasted in multiple locations such as Amsterdam, Rio de Janeiro and Sydney, the user's request will be sent to Amsterdam.
The routing decision is made by \gls{bgp} which directs the request to a single \gls{pop}, typically one that is topologically close to the user~\cite{rfc4786} from a routing, and often geographically~\cite{remiservillo-2025}, perspective.

A common misconception is that announcing an address at multiple locations, in itself, is classified as anycast.
Many \glspl{cdn} announce their prefixes at multiple network edges (in different geographic regions), where they route traffic internally to a single \gls{pop} hosting the service.
Critically, as defined in RFC~4786~\cite{rfc4786}, anycast is the replication of a service address where several \glspl{pop} make the service available.
% G: However, full replication is not always true (see chapter anycast).

There are multiple reasons for providing a service using anycast.
Anycast can increase service resilience through automatic failover.
% G: text from paper: (where traffic is redirected to a different \gls{pop} if one fails).
For instance, in our previous example, if the Amsterdam \gls{pop} is unavailable, the user's request would be redirected to the next best \gls{pop} (either Rio de Janeiro or Sydney, depending on routing decisions).
Furthermore, anycast offers service replication closer to users to reduce latency, and enables load distribution across \glspl{pop}.

\section{TLS}
\label{sec:background-tls}
The \acrfull{tls} is a protocol that provides secure communications over the network via a set of cryptographic algorithms and certificate-based authentication.
Historically, \gls{tls} was introduced in 1994 with \gls{ssl} version 1.0~\cite{_WhatSSLSecure_} and 
\gls{ietf} standardized \gls{tls} in 1999 and the current version is 1.3~\cite{rfc8446}, while 1.2 ~\cite{rfc5246} is still widely used and not considered deprecated.
% https://www.cs.auckland.ac.nz/courses/compsci726s2c/lectures/ra-2021/COMPSCI726-SSL-TLS.pdf
Examples of known application-layer protocols that use \gls{tls} are HTTPS to communicate with web servers, SMTPS for email servers and LDAPS for directory services.

\gls{tls} was designed with the goal of providing confidentiality, integrity, and authentication.
To achieve these properties, client and server have a handshake phase before setting up the secure communication channel.
We briefly discuss the \gls{tls} handshake and how it achieves these properties.

\subsection{TLS handshake}
The \gls{tls} handshake occurs right after a \gls{tcp} connection is established between a client and a server.
We depict a simplified view of the \gls{tls} handshake in~\cref{fig:background-tls-handshake}, where we explain the key steps and ideas.
We refer to the literature for other messages and details necessary for implementation of the protocol~\cite{rfc5246,rfc8446}.

The \gls{tls} handshake starts with the client sending a \texttt{ClientHello} message.
This message contains the client's latest (higher valued) supported \gls{tls} version, session ID, cipher suites supported (from the strongest to the weakest), compression methods (used to compress data prior to encryption) it supports and a random value.
The session ID is optional and when present, allows the server to resume a previous session reusing \gls{tls} configurations and keys from a server-cached handshake and the handshake proceeds directly to the \texttt{ChangeCipherSpec} and \texttt{Finished} messages.
Otherwise, a new session ID is generated by the server and sent in the \texttt{ServerHello} message.
The client random values is used to generate the master secret alongside the server's random value from the \texttt{ServerHello} message.

The server replies with a \texttt{ServerHello} message that contains the selected \gls{tls} version, a random value, session ID, selected cipher suite, the selected compression method and optionally extensions (\eg, the \gls{sni}~\cite{rfc6066}).
The server chooses the best (most secure) \gls{tls} version and cipher suite which are supported by both parties.
In case the parties do not agree on a common \gls{tls} version or cipher suite, the handshake fails, alert messages are exchanged between the client and server and the connection is closed.
We provide more details about \gls{tls} cipher suites in~\cref{subsec:background-tls-ciphers}.

The server sends its X.509 certificate first (leaf or end-host certificate) and a chain of certificates in the \texttt{Certificate} message.
The distribution of certificates, its content and its validation is briefly discussed in~\cref{subsec:background-tls-pki}.
The client uses the certificates to authenticate the server.
For instance, by validating the chain, and the validity period of the certificates, and most importantly, by verifying the signature of the leaf certificate using the public key of the server.
The signature is formed from the encryption of the hash of the certificate contents using the certificate issuer's private key.
The client can compute the digest and verify against the decrypted signature using the server's public key.
This process ensures that the certificate is not tampered and that the client is communicating with the intended server.

Client and server perform the key exchange process.
There are different key exchange methods, such as \gls{rsa}, \gls{dh} and \gls{ecdh}, where the last two can be used for ephemeral key exchange.
The key exchange allows client and server to generate a premaster secret that is used to derive the session keys.
The premaster secret is trasmitted either by encrypting using the server's public key (in \gls{rsa}) or by using the \gls{dh} key exchange method to generate the secret without transmitting it.
The public parameters of the \gls{dh} can be statically defined in the certificate~\cite{rfc5114} or generated on every session, which is called ephemeral.
Ephemeral key exchange methods provide \gls{pfs}, which means that even if the server's private key is compromised, past communications remain secret.
The session key is used to encrypt and decrypt the application data during the \gls{tls} session.
% https://security.stackexchange.com/questions/94390/whats-the-purpose-of-dh-parameters

Both client and server send a \texttt{ChangeCipherSpec} message to indicate that subsequent messages will be encrypted using the negotiated cipher suite and session keys.
Finally, both parties exchange a \texttt{Finished} message to verify that the handshake was successful.
The \texttt{Finished} message contains a hash of all handshake messages, which is encrypted using the session key.
This approach ensures the integrity of the handshake.

% handshake steps 1.3
The modern \gls{tls} version 1.3 handshake consists of fewer steps.
For instance, the \texttt{ClientHello} message already contains a pre-shared key (either externally obtained or from previous handshake).
The \texttt{ServerHello} message contains the certificates, the pre-shared key and early data.
After that, the server immediately sends the \texttt{Finished} message, and the client computes the verifications and sends its \texttt{Finished} message.
The \gls{tls} v1.3 only uses ephemeral key exchange methods.
Furthermore, the \gls{tls} v1.3 handshake allows for 0-\gls{rtt} handshake, which means that the client can send early application data in the first message to the server, using the computed session key.

\subsection{TLS and the PKI}
\label{subsec:background-tls-pki}
We briefly describe the structure of X.509 certificates and the public key infrastructure (PKI) used to authenticate servers during the TLS handshake.
X.509 certificates follow the hierarchical model defined in X.500 and are standardized by ITU-T~\cite{itu-t-x509}.
The X.509 certificates were created to authenticate users to read and write information in directory services~\cite{holz-dissertation}.
Over time, X.509 certificates have been widely adopted in applications such as S/MIME~\cite{OndaroEtAl_MINECollectingAnalyzing_2025} and services secured with \gls{tls}, including Web and LDAP~\cite{gustavo-ldap2}, with the Web driving their large-scale deployment~\cite{holz2020}.

X.509 certificates contain the following fields~\cite{itu-t-x509}: the version (typically version 3), the serial number (unique unique per issuing \gls{ca}), the Subject and Issuer information (\eg, Common Name, Organization name and Organization Unit), the subject public key with the algorithm identifier and the key, the validity period of the certificate (\textit{not before} and \textit{not after}), and the signature algorithm identifier and signature of the certificate authority that issued the certificate.
X.509 certificates may include optional extensions, where the most common and relevant to this thesis are: \gls{san} and the \textit{CA} flag.
For instance, the \gls{san} extension specifies additional identities associated with the certificate, such as domain names or IP addresses.
The \textit{CA} flag extension, which when set to ``true'', indicates that the certificate can be used to sign other certificates.
In chapter (TODO ref LDAP) we show that \gls{ldap} servers use broken chains of certificates with the \textit{CA} flag set to ``true'' in the leaf certificates.

X.509 certificates follow a hierarchical structure that enables a chain of trust.
For instance, a leaf certificate is usually signed by an intermediate certificate, until reaching a trusted root certificate.
A chain can have multiple intermediate certificates in the chain, or the leaf can be directly signed by a root certificate.
The intermediate and root certificates are issued by a trusted \gls{ca}.
The \gls{ca} is responsible for verifying the identity of certificate requesters.
Hence, \gls{ca} are critical entities in the \gls{pki} ecosystem., and reaching such level of trust is a complex process from well-audited forums such as the CA/Browser Forum~\cite{cabr}.
When a \gls{ca} goes rogue, it can issue fraudulent certificates.
As a consequence, rogue \glspl{ca} are removed from the trusted root store, such as the case of DigiNotar removed from browsers~\cite{NightingaleDigiNotarRemovalFollow2011}.
Trusted root stores are maintained and embedded in the \glspl{os} (\eg, Linux, Apple, and Windows), browsers (\eg, Chrome), or maintained by other entities (\eg, \gls{ccadb}\footnote{\url{ccadb.org}}).
Certificates can also be self-signed, however their use is generally discouraged and limited to specific scenarios (see~\cref{subsec:background-ldap-pki}).
When certificates are signed by a \gls{ca}, it creates a cryptographic binding between the identity of parent and child certificates.
This binding enables clients to verify the authenticity of certificates during the \gls{tls} handshake.

Certificate validity depends on several conditions.
For instance, valid certificates have the chain of trust from the leaf certificate cryptographically leading to a publicly and trusted root certificate.
Additionally, the validity period by the time of checking must be within the \textit{not before} and \textit{not after} fields.
Furthermore, the signature of the leaf certificate must match the issuer's public key, and the signature algorithm should be secure (\eg, SHA-256)~\cite{X509certinsecuresigningalgorithmBoostSecurity}.
Hostname verification may additionally be performed, where the domain name of the server must match the Common Name or the \gls{san} field of the certificate.
Finally, the revocation status of a certificate can be checked using mechanisms such as \gls{crl}~\cite{rfc5280} or the \gls{ocsp}~\cite{rfc6960}.

\section{LDAP}
\label{sec:background-ldap}
% copy from ldap-1 and ldap-2
Similar to X.509 certificates, \gls{ldap} is also derived from the X.500 standard.
\acrfull{ldap} is a client-server protocol to facilitate transfer and access of data stored in directory services such as OpenLDAP or \acrfull{ad}.
We use the generic term \textit{\gls{ldap} servers} to refer to all servers that speak the protocol and distinguish by directory service as needed.
\gls{cldap} runs over UDP and it is restricted to a small set of queries.
In this thesis we do not investigate.
\gls{ldap} typically runs over TCP on port 389 (clear text with optional \starttls) or 636 (implicit \gls{tls}), while ports 3268 (clear text/\starttls) and 3269 (implicit \gls{tls}) are reserved for the \gls{gc} service in Microsoft's \gls{ad}.
The \gls{gc} is an \gls{ad} role to manage and access copies of data items across a multitude of \gls{ldap} servers~\cite{alvinashcraft_GlobalCatalogWin32_2020}.
\gls{ldap} stores data in a tree-like hierarchy.
The special Root \gls{dse} data entry stores information about the server's functionality in form of attributes.
These attributes with a predefined type and a value.
The type determines the syntax, semantics, and characteristics of the value.
Attribute types are uniquely identified internationally by their \gls{oid}~\cite{x500standard,x501Standard}.
These attributes hold a unique significance, as they encompass essential information about the server, such as supported controls, extensions, features, \gls{ldap} versions, authentication mechanism such as \gls{sasl}, and more~\cite{LDAP_RFC_Model}, which can help identify the \gls{ldap} product.

\gls{ldap} may use \gls{tls}, where a cipher suite is agreed upon and a chain of X.509 certificates is sent to a client.
The RFC-9325 gives best-practice recommendations.
% use case of LDAP

\subsection{LDAP Authentication and Authorization}
Clients authenticate to an LDAP server by attempting a bind operation. 
A connection between the client and the server is established if the bind is successful~\cite{rfc4511, LDAP_RFC_AUTHENTICATION}.
As part of the bind request, the client chooses an authentication method and, if necessary, supplies authentication credentials.
LDAPv3 supports four types of authentication: simple, anonymous, unauthenticated, and \gls{sasl} authentication~\cite{LDAP_RFC_AUTHENTICATION}.

The simple authentication uses an username and a cleartext password.
This method requires \gls{tls} to encrypt the password.
Anonymous authentication uses empty username and password.
% TODO Why is this used?
Furthermore, the unauthenticated authentication uses an username but without a password.
It's worth noting that RFC 4513~\cite{LDAP_RFC_AUTHENTICATION} recommends disabling this method by default for clients and servers.
This method can be used with \gls{sso} to retrieve the \gls{dn} from the directory to proceed with the authentication.
Finally, \gls{sasl} supports a wide range of authentication methods, such as username/password, Kerberos, and digital certificates.
Anonymous and plaintext \gls{sasl} authentication is not commonly used in \gls{ldap}~\cite{rfc4513}.
While some are notably robust, others have been identified as insecure~\cite{RFC4422SASL}.

\subsection{LDAP PKI}
\label{subsec:background-ldap-pki}
% copy from ldap-2
A certificate chain must be anchored at a trusted root, requiring the \gls{ldap} client to be configured with the required root certificate(s).
The so-called end-host (or leaf) certificate provides a cryptographic binding of an identifier---typically the domain name of the \gls{ldap} server---to a corresponding public key.
On the Web, public \glspl{ca} issue certificates under CA/Browser Forum rules (\eg, verification of domain ownership).
%, and root stores are managed by the \gls{os} (\eg, Windows, macOS, and Linux) and browser vendors (\eg, Firefox).
For \gls{ldap}, no comparable body or standardization efforts exist.
In many cases (\eg, Linux, Windows), the root store that the \gls{ldap} client relies on is the same as for the Web.
In refined setups, the \gls{ldap} server administrator may supply it directly, even using a private \gls{ca} to issue their own certificates.
In this case, the administrators must also deploy the root certificate to their \gls{ldap} clients using tools like \gls{ad} Certificate Services or scripts.
In such a \textit{private \gls{pki}}, one should issue intermediate certificates from the root certificate, which in turn are used to issue a leaf certificate.
This setup is important as it allows the private key for the root certificate to be kept in a secure location. An alternative, semi-private \gls{pki} setup is also possible: an operator could request a sub-\gls{ca} certificate from a public \gls{ca}, \ie, an intermediate certificate, from which they then issue their own certificates.
This setup must be tightly controlled so leaf certificates are only ever issued for domains under control of the administrator~\cite{cabr}.
% https://cabforum.org/working-groups/server/baseline-requirements/requirements/ 3.2.2.4.4

A discouraged practice is to deploy so-called self-signed certificates, where the key used to sign a certificate is also the certified key.
In such a case, the client must be pre-configured to accept only this certificate---or users told to ignore the warnings their \gls{ldap} client would show.
The setup is only suitable if the \gls{ldap} server serves a minimal set of users.
It also constitutes a security risk as the private key remains online in memory.
As certificates are available for free and can be relatively easily deployed with the ACME protocol, there is little justification to use self-signed certificates any more.

\section{Email Security Protocols}
\label{sec:background-dmarc}
In this section, we provide background information on email security protocols relevant to this thesis (\gls{spf}, \gls{dkim}, and \gls{dmarc}) and how \gls{dmarc} interact with them during email communications.

\subsection{DKIM}
\acrfull{dkim} is an email authentication and integrity verification method that employs public-key cryptography. Domain owners use \gls{dkim} to digitally sign emails from their domain. During the \gls{smtp} communication, the sender adds the \texttt{DKIM-Signature} header field with \gls{dkim} relevant tags. For example, the signing domain (``d'' tag), a key selector (``s'' tag), and a signature (signed with the sender's private key, ``b'' tag).
The recipient's email server queries the sender's \gls{dns} server for its TXT record (\eg, \textit{<s>.\_domainkey.<d>.example.com}) to retrieve the public key, and then uses it to verify the received email signature. The verification results can be either \textit{pass}, \textit{permfail}, or \textit{tempfail}. Multiple signatures can be present in an email, and if at least one is valid, \gls{dkim} evaluation succeeds.
% https://easydmarc.com/blog/what-are-dkim-tags/
% https://dmarcly.com/blog/how-to-implement-dmarc-dkim-spf-to-stop-email-spoofing-phishing-the-definitive-guide#what-is-dkim

\subsection{SPF}
\acrfull{spf} allows a domain owner to specify a list of IP addresses that are authorized to send emails on behalf of their domain.
The domain owner configures the \gls{spf} rules through the \gls{dns} TXT record.
The recipient's mail server extracts the IP address and the domain name from ``MAIL FROM'' \textbf{or} ``HELO/EHLO FROM'' identify during the \gls{smtp} communication.
\gls{spf} verification is done through the \textbf{check\_host()} function~\cite{rfc7208}, which uses the identities and the sender's IP address from the \gls{smtp} communication, and retrieves the \gls{spf} rules to determine whether the sender's IP address belongs to the list of IP addresses configured.
The outcome of the verification can be \textit{none} (no conclusion about authorization), \textit{neutral} (treated as ``none''), \textit{pass}, \textit{fail}, \textit{softfail} (requires further investigation), \textit{temperror} (transient error during verification), or \textit{permerror} (incorrect \gls{spf} records).
The recipient's mail server can then decide to accept, reject, or mark the email as spam based on the verification result and their own anti-spoofing policies.
\gls{dmarc} uses results of the \textbf{check\_host()} function based on ``MAIL FROM'' identity.

\subsection{DMARC}
\acrfull{dmarc} data is published in the \gls{dns} TXT records of the \textit{\_dmarc} subdomain (\eg, \textit{\_dmarc.example.com}).
The \gls{dmarc} records can be as follows:

\begin{center}
    \texttt{v=DMARC1; p=reject; rua=mailto:rua@example.com}
\end{center}

The record always starts with the tag \texttt{v=DMARC1} followed by a list of tags separated by semicolons.
The tags relevant this thesis are:
\begin{itemize}
    \item[p] Policy for organizational domain. It tells the receiving mail server how to handle emails that fail \gls{dmarc} verification. The possible values are \texttt{none} (monitoring only), \texttt{quarantine}, and \texttt{reject} (block delivery).
    \item[sp] Similar to \texttt{p}, but for subdomains. If \texttt{sp} is not specified, the value of \texttt{p} tag is used. If the subdomain has its own \texttt{p} tag, then it takes precedence over the \texttt{sp} tag.
    \item[rua] Specify the email address(es) to which ``aggregate reports'' (see below) should be sent with ``mailto:address''. Multiple addresses can be specified.
    \item[ruf] Similar to \texttt{rua}, but for ``forensic reports'' (see below).
\end{itemize}

\gls{dmarc} uses verification results from \gls{spf} and \gls{dkim} and also domain name alignment to determine the authenticity of an email.
The \gls{spf} domain name alignment occurs when the ``FROM'' domain name matches the ``MAIL FROM'' domain name of the received email.
The \gls{dkim} alignment occurs when the domain name present in the ``d'' tag matches the ``MAIL FROM'' domain name.
\gls{dmarc} verification passes if \gls{spf} check passes and the \gls{spf} domain name is aligned \textbf{or} \gls{dkim} check passes and the \gls{dkim} domain name is aligned.

%\textbf{\gls{dmarc} reporting}
\gls{dmarc} sends two kinds of reports: \textbf{Aggregate} (in \texttt{rua} tag) and \textbf{Forensic} (in \texttt{ruf} tag)~\cite{rfc7489}. Aggregate reports provide a summary, in XML format, of \gls{dmarc}, \gls{spf}, and \gls{dkim} results (pass and fail), the sending IP address, \textit{RFC5322.From} address, etc.
The use of forensic reports is limited due to privacy concerns~\cite{hureau_stress_testing,_BSITR03182Email_}.
Reports help domain owners understand their domain's email usage, such as tracking policy effectiveness and spotting abuse or misconfigurations.
The nature of forensic reports can lead to parties that receive such reports to have access to sensitive information, although previous work has shown that such reports are falling short of use~\cite{hureau_stress_testing}.
\gls{dmarc} reports are often sent to another domain (\eg, a specialized organization) for processing, as handling raw reports is complex~\cite{_DMARCReportsGoogle_}.

\subsection{DMARC role in email communications}
~\cref{fig:background-dmarc} depicts a simplified overview of the \gls{dmarc} role in email communication.
Bob (sending domain) sends an email to Alice (receiving domain).
Alice queries Bob's \gls{dns} server to retrieve \gls{dkim}, \gls{spf}, and \gls{dmarc} records to verify the email's authenticity and decide whether to deliver it.
When a \texttt{rua} address is present, Alice's email server is requested to report to Bob.
The reports provide insights into how Bob's domain is used and whether it is legitimate.
