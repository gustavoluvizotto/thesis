\section{Anycast}\label{sec:background-anycast}
The first RFC describing anycast (RFC~1546~\cite{rfc1546}) was published in 1993 and explains how anycast simplifies finding an appropriate server.
As service examples, they list DNS, FTP, and ``Archie'' (one of the first Internet search engines).
Nowadays, anycast is widely deployed for critical Internet infrastructure, such as the DNS~\cite{dns-anycast}, or by \glspl{cdn} to provide web services~\cite{CicaleseEtAl_CharacterizingIPv4Anycast_2015, GiordanoEtAl_FirstCharacterizationAnycast_}.

Anycast is the practice of making a service IP address (either IPv4 or IPv6) available in multiple physical locations~\cite{rfc4786}.
Such locations are referred to catchments or simply as \glspl{pop}.
% G: \glspl{pop} are also catchments. Not needed to introduce here.
For instance, when a user is located in Enschede trying to access a website that is anycasted in multiple locations such as Amsterdam, Rio de Janeiro and Sydney, the user's request will be sent to Amsterdam.
The routing decision is made by \gls{bgp} which directs the request to a single \gls{pop}, typically one that is topologically close to the user~\cite{rfc4786} from a routing, and often geographically~\cite{remiservillo-2025}, perspective.

A common misconception is that announcing an address at multiple locations, in itself, is classified as anycast.
Many \glspl{cdn} announce their prefixes at multiple network edges (in different geographic regions), where they route traffic internally to a single \gls{pop} hosting the service.
Critically, as defined in RFC~4786~\cite{rfc4786}, anycast is the replication of a service address where several \glspl{pop} make the service available.
% G: However, full replication is not always true (see chapter anycast).

There are multiple reasons for providing a service using anycast.
Anycast can increase service resilience through automatic failover.
% G: text from paper: (where traffic is redirected to a different \gls{pop} if one fails).
For instance, in our previous example, if the Amsterdam \gls{pop} is unavailable, the user's request would be redirected to the next best \gls{pop} (either Rio de Janeiro or Sydney, depending on routing decisions).
Furthermore, anycast offers service replication closer to users to reduce latency, and enables load distribution across \glspl{pop}.

\section{TLS}
\label{sec:background-tls}
The \acrfull{tls} is a protocol that provides secure communications over the network via a set of cryptographic algorithms and certificate-based authentication.
Historically, \gls{tls} was introduced in 1994 with \gls{ssl} version 1.0~\cite{_WhatSSLSecure_} and 
\gls{ietf} standardized \gls{tls} in 1999 and the current version is 1.3~\cite{rfc8446}, while 1.2 ~\cite{rfc5246} is still widely used and not considered deprecated.
% https://www.cs.auckland.ac.nz/courses/compsci726s2c/lectures/ra-2021/COMPSCI726-SSL-TLS.pdf
Examples of known application-layer protocols that use \gls{tls} are HTTPS to communicate with web servers, SMTPS for email servers and LDAPS for directory services.

\gls{tls} was designed with the goal of providing confidentiality, integrity, and authentication.
To achieve these properties, client and server have a handshake phase before setting up the secure communication channel.
We briefly discuss the \gls{tls} handshake and how it achieves these properties.

\subsection{TLS handshake}

The \gls{tls} handshake starts with the client sending a \texttt{ClientHello} message.
This message contains the client's latest (higher valued) supported \gls{tls} version, session ID, cipher suites supported (from the strongest to the weakest), compression methods (used to compress data prior to encryption) it supports and a random value.
The session ID is optional and when present, allows the server to resume a previous session reusing \gls{tls} configurations and keys from a server-cached handshake and the handshake proceeds directly to the \texttt{ChangeCipherSpec} and \texttt{Finished} messages.
Otherwise, a new session ID is generated by the server and sent in the \texttt{ServerHello} message.
The client random values is used to generate the master secret alongside the server's random value from the \texttt{ServerHello} message.

The server replies with a \texttt{ServerHello} message that contains the selected \gls{tls} version, a random value, session ID, selected cipher suite, the selected compression method and optionally extensions (\eg, the \gls{sni}~\cite{rfc6066}).
The server chooses the best (most secure) \gls{tls} version and cipher suite which are supported by both parties.
In case the parties do not agree on a common \gls{tls} version or cipher suite, the handshake fails, alert messages are exchanged between the client and server and the connection is closed.
We provide more details about \gls{tls} cipher suites in~\cref{subsec:background-tls-ciphers}.

The server sends its X.509 certificate first (leaf or end-host certificate) and a chain of certificates in the \texttt{Certificate} message.
The distribution of certificates, its content and its validation is briefly discussed in~\cref{subsec:background-tls-pki}.
The client uses the certificates to authenticate the server.
For instance, by validating the chain, and the validity period of the certificates, and most importantly, by verifying the signature of the leaf certificate using the public key of the server.
The signature is formed from the encryption of the hash of the certificate contents using the certificate issuer's private key.
The client can compute the digest and verify against the decrypted signature using the server's public key.
This process ensures that the certificate is not tampered and that the client is communicating with the intended server.

The server sends a \texttt{ServerHelloDone} message to indicate that the server waits for the key exchange process to begin.
After that, client and server perform the key exchange process.
There are different key exchange methods, such as RSA, Diffie-Hellman and Elliptic Curve Diffie-Hellman, where the last two can be used for ephemeral key exchange.
The key exchange allows client and server to generate a shared secret that is used to derive the session keys.
The session key is used to encrypt and decrypt the application data during the \gls{tls} session.
The shared secret can be reused or be regenerated for each session.
Ephemeral key exchange methods provide perfect forward secrecy, which means that even if the server's private key is compromised, past communications remain secret.
% TODO check RFC for more details to add

Then both client and server send a \texttt{ChangeCipherSpec} message to indicate that subsequent messages will be encrypted using the negotiated cipher suite and session keys.
Finally, both parties exchange a \texttt{Finished} message to verify that the handshake was successful and that both parties have the same session keys.
The \texttt{Finished} message contains a hash of all handshake messages, which is encrypted using the session key.
This approach ensures the integrity of the handshake.

\section{Email Security Protocols}
\label{sec:background-dmarc}
In this section, we provide background information on email security protocols relevant to this thesis (\gls{spf}, \gls{dkim}, and \gls{dmarc}) and how \gls{dmarc} interact with them during email communications.

\subsection{DKIM}
\acrfull{dkim} is an email authentication and integrity verification method that employs public-key cryptography. Domain owners use \gls{dkim} to digitally sign emails from their domain. During the \gls{smtp} communication, the sender adds the \texttt{DKIM-Signature} header field with \gls{dkim} relevant tags. For example, the signing domain (``d'' tag), a key selector (``s'' tag), and a signature (signed with the sender's private key, ``b'' tag).
The recipient's email server queries the sender's \gls{dns} server for its TXT record (\eg, \textit{<s>.\_domainkey.<d>.example.com}) to retrieve the public key, and then uses it to verify the received email signature. The verification results can be either \textit{pass}, \textit{permfail}, or \textit{tempfail}. Multiple signatures can be present in an email, and if at least one is valid, \gls{dkim} evaluation succeeds.
% https://easydmarc.com/blog/what-are-dkim-tags/
% https://dmarcly.com/blog/how-to-implement-dmarc-dkim-spf-to-stop-email-spoofing-phishing-the-definitive-guide#what-is-dkim

\subsection{SPF}
\acrfull{spf} allows a domain owner to specify a list of IP addresses that are authorized to send emails on behalf of their domain.
The domain owner configures the \gls{spf} rules through the \gls{dns} TXT record.
The recipient's mail server extracts the IP address and the domain name from ``MAIL FROM'' \textbf{or} ``HELO/EHLO FROM'' identify during the \gls{smtp} communication.
\gls{spf} verification is done through the \textbf{check\_host()} function~\cite{rfc7208}, which uses the identities and the sender's IP address from the \gls{smtp} communication, and retrieves the \gls{spf} rules to determine whether the sender's IP address belongs to the list of IP addresses configured.
The outcome of the verification can be \textit{none} (no conclusion about authorization), \textit{neutral} (treated as ``none''), \textit{pass}, \textit{fail}, \textit{softfail} (requires further investigation), \textit{temperror} (transient error during verification), or \textit{permerror} (incorrect \gls{spf} records).
The recipient's mail server can then decide to accept, reject, or mark the email as spam based on the verification result and their own anti-spoofing policies.
\gls{dmarc} uses results of the \textbf{check\_host()} function based on ``MAIL FROM'' identity.

\subsection{DMARC}
\acrfull{dmarc} data is published in the \gls{dns} TXT records of the \textit{\_dmarc} subdomain (\eg, \textit{\_dmarc.example.com}).
The \gls{dmarc} records can be as follows:

\begin{center}
    \texttt{v=DMARC1; p=reject; rua=mailto:rua@example.com}
\end{center}

The record always starts with the tag \texttt{v=DMARC1} followed by a list of tags separated by semicolons.
The tags relevant this thesis are:
\begin{itemize}
    \item[p] Policy for organizational domain. It tells the receiving mail server how to handle emails that fail \gls{dmarc} verification. The possible values are \texttt{none} (monitoring only), \texttt{quarantine}, and \texttt{reject} (block delivery).
    \item[sp] Similar to \texttt{p}, but for subdomains. If \texttt{sp} is not specified, the value of \texttt{p} tag is used. If the subdomain has its own \texttt{p} tag, then it takes precedence over the \texttt{sp} tag.
    \item[rua] Specify the email address(es) to which ``aggregate reports'' (see below) should be sent with ``mailto:address''. Multiple addresses can be specified.
    \item[ruf] Similar to \texttt{rua}, but for ``forensic reports'' (see below).
\end{itemize}

\gls{dmarc} uses verification results from \gls{spf} and \gls{dkim} and also domain name alignment to determine the authenticity of an email.
The \gls{spf} domain name alignment occurs when the ``FROM'' domain name matches the ``MAIL FROM'' domain name of the received email.
The \gls{dkim} alignment occurs when the domain name present in the ``d'' tag matches the ``MAIL FROM'' domain name.
\gls{dmarc} verification passes if \gls{spf} check passes and the \gls{spf} domain name is aligned \textbf{or} \gls{dkim} check passes and the \gls{dkim} domain name is aligned.

%\textbf{\gls{dmarc} reporting}
\gls{dmarc} sends two kinds of reports: \textbf{Aggregate} (in \texttt{rua} tag) and \textbf{Forensic} (in \texttt{ruf} tag)~\cite{rfc7489}. Aggregate reports provide a summary, in XML format, of \gls{dmarc}, \gls{spf}, and \gls{dkim} results (pass and fail), the sending IP address, \textit{RFC5322.From} address, etc.
The use of forensic reports is limited due to privacy concerns~\cite{hureau_stress_testing,_BSITR03182Email_}.
Reports help domain owners understand their domain's email usage, such as tracking policy effectiveness and spotting abuse or misconfigurations.
The nature of forensic reports can lead to parties that receive such reports to have access to sensitive information, although previous work has shown that such reports are falling short of use~\cite{hureau_stress_testing}.
\gls{dmarc} reports are often sent to another domain (\eg, a specialized organization) for processing, as handling raw reports is complex~\cite{_DMARCReportsGoogle_}.

\subsection{DMARC role in email communications}
~\cref{fig:background-dmarc} depicts a simplified overview of the \gls{dmarc} role in email communication.
Bob (sending domain) sends an email to Alice (receiving domain).
Alice queries Bob's \gls{dns} server to retrieve \gls{dkim}, \gls{spf}, and \gls{dmarc} records to verify the email's authenticity and decide whether to deliver it.
When a \texttt{rua} address is present, Alice's email server is requested to report to Bob.
The reports provide insights into how Bob's domain is used and whether it is legitimate.
